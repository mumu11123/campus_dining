# 二、数据库总体设计

## 2.1 数据库名称

```sql
campus_dining
```

---

## 2.2 数据库设计原则

本系统数据库设计遵循以下原则：

### （1）统一命名规范

统一采用：

- 小写英文
- 下划线命名法（snake_case）

例如：

```sql
created_at
updated_at
order_item
```

避免：

```sql
createTime
CreateTime
orderItem
```

等混乱命名。

---

### （2）统一主键规范

所有核心表均采用：

```sql
BIGINT AUTO_INCREMENT
```

作为主键。

优点：

- 支持大规模数据增长
- 与 Java Long 类型对应
- 后续方便分库分表

---

### （3）统一字符集

全系统采用：

```sql
utf8mb4
```

支持：

- 中文
- Emoji
- 多语言字符

避免乱码问题。

---

### （4）统一时间字段

所有核心表统一包含：

```sql
created_at
updated_at
```

用于：

- 数据追踪
- 日志分析
- 审计记录
- 排查问题

---

### （5）软删除机制

核心业务表统一增加：

```sql
is_deleted
```

实现逻辑删除。

原因：

- 防止误删数据
- 保留历史记录
- 方便数据恢复
- 满足审计需求

例如：

```text
0 正常
1 已删除
```

# 4.1 学生用户表（user）

## 功能

用于存储学生账号信息。

---

## 核心字段

| 字段 | 说明 |
|---|---|
| username | 登录账号 |
| password | BCrypt 加密密码 |
| real_name | 真实姓名 |
| phone | 手机号 |
| avatar | 用户头像 |
| status | 用户状态 |
| is_deleted | 软删除 |

---

## 设计特点

### （1）账号唯一

```sql
UNIQUE KEY uk_username
```

防止重复注册。

---

### （2）手机号唯一

保证：

- 一个手机号只能绑定一个账号

---

### （3）软删除机制

用户注销后：

不会真正删除数据。

避免：

- 订单丢失
- 历史记录异常

---

# 4.2 商家表（merchant）

## 功能

存储审核通过后的商家信息。

---

## 核心字段

| 字段 | 说明 |
|---|---|
| shop_name | 店铺名称 |
| category | 店铺分类 |
| business_license | 营业执照 |
| status | 营业状态 |

---

## 商家状态设计

| 状态值 | 含义 |
|---|---|
| 1 | 正常营业 |
| 2 | 暂停营业 |
| 3 | 禁止营业 |

---

## 设计亮点

### （1）商家与申请分离

本系统：

- merchant 表只存正式商家
- merchant_apply 存申请记录

优点：

- 业务逻辑清晰
- 支持多次申请
- 支持历史审核记录保留

---

### （2）支持扩展商家资料

预留：

- logo
- description
- business_license

便于后续：

- 店铺装修
- 商家认证
- 平台展示

---

# 4.3 商家入驻申请表（merchant_apply）

## 功能

记录商家入驻申请流程。

---

## 设计原因

如果直接在 merchant 表审核：

会导致：

- 未通过商家混入正式数据
- 无法保存审核历史

因此单独设计：

```sql
merchant_apply
```

实现：

- 独立审核生命周期

---

## 审核状态

| 状态值 | 含义 |
|---|---|
| 0 | 待审核 |
| 1 | 已通过 |
| 2 | 已驳回 |
| 3 | 退回补充 |

---

## 审核流程

```text
商家提交申请
        ↓
管理员审核
        ↓
审核通过 → 创建 merchant 数据
审核失败 → 保留申请记录
```

# 订单状态流转设计

本系统采用：

完整订单生命周期设计。

---

## 状态流转

```text
待支付
   ↓
已支付
   ↓
待接单
   ↓
制作中
   ↓
待取餐
   ↓
已完成
```

异常情况：

```text
待支付 → 已取消
待接单 → 已取消
```

---

## 状态值定义

| 值 | 状态 |
|---|---|
| 0 | 待支付 |
| 1 | 已支付 |
| 2 | 待接单 |
| 3 | 制作中 |
| 4 | 待取餐 |
| 5 | 已完成 |
| 6 | 已取消 |

---

## 订单编号设计

字段：

```sql
order_no
```

采用：

```text
Redis 自增
或
雪花算法
```

生成。

原因：

- 保证唯一性
- 避免并发冲突
- 支持高并发订单

---

## 金额字段设计

| 字段 | 作用 |
|---|---|
| total_amount | 原总金额 |
| discount_amount | 优惠金额 |
| actual_amount | 实付金额 |

---

## 设计亮点

### 金额拆分设计

便于：

- 满减
- 优惠券
- 折扣活动

扩展。

# 4.8 订单明细表（order_item）

## 功能

记录订单中的具体菜品。

---

## 订单快照机制

系统保存：

- 下单时菜品名称
- 下单时价格

而不是实时读取 dish 表。

---

## 原因

如果后续：

- 商家改价
- 商家改名

历史订单不会被影响。

---

## 示例

```text
下单时：
红烧肉 18元

后续改价：
红烧肉 22元

历史订单仍显示：
18元
```

---

# 4.9 聊天消息表（message）

## 功能

- 学生 ↔ 商家
- 学生 ↔ 客服
- 商家 ↔ 管理员

---

## 设计特点


- user
- merchant
- admin

因此：

不能使用传统物理外键。

采用：

```text
逻辑外键
```

设计。

---

### （2）支持订单关联

通过：

```sql
order_id
```

实现：

订单聊天记录追踪。
